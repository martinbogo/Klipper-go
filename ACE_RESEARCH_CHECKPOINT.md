# Anycubic ACE Pro Protocol - Reverse Engineering Checkpoint

## Objective
Establish a stable, independent connection to the Anycubic ACE hardware via USB Serial (`/dev/ttyACM0`) without relying on the proprietary `gklib` daemon, enabling custom Klipper integration and control.

## Hardware & Connection Behavior
- **Device:** Anycubic ACE (Multi-Color/Material system).
- **Interface:** USB CDC ACM (`/dev/ttyACM0` on the printer host).
- **Aggressive Watchdog:** The ACE hardware has a strict software-level watchdog. If an application opens the serial port but does not provide a valid, correctly framed handshake payload almost immediately, the ACE device simulates a physical USB disconnect. 
  - Host kernel logs will show `USB disconnect` and software will throw `input/output error`.
  - This is NOT related to hardware DTR/RTS pin toggling (which was isolated via raw `TCGETS`/`TCSETS` `unix` syscalls with `CLOCAL` flag to ignore modem lines).

## Serial Port Configuration
- **Baud Rate:** 115200
- **Mode:** Raw (`O_RDWR|O_NOCTTY|O_NONBLOCK`)
- **Flags:** `CS8 | CREAD | CLOCAL`
- **Crucial:** Using raw UNIX termios syscalls bypasses higher-level libraries (like `tarm/serial`) that might implicitly pulse DTR/RTS or send `break` signals, ensuring exact parity with the manufacturer's initialization process.

## The V1 Communication Protocol
We discovered that the initial interaction by Klipper/`gklib` is not a binary ping/heartbeat, but rather framed JSON.

### Packet Structure
Packets follow this binary envelope:
1. `0xFF` (Frame Start 1)
2. `0xAA` (Frame Start 2)
3. `Len_Low` (Payload length, Little Endian)
4. `Len_High` (Payload length, Little Endian)
5. `... JSON String Payload ...` (e.g., `{"id":2,"method":"get_info"}`)
6. `CRC_Low` (Little Endian)
7. `CRC_High` (Little Endian)
8. `0xFE` (Frame End)

### Checksum (CRC)
The CRC is calculated **only over the JSON payload bytes**, not the frame headers or length.
Go implementation derived from `project/extras_ace_commun.go`:
```go
func _calc_crc(buf []byte) uint16 {
	var _crc uint16 = 0xffff
	for i := 0; i < len(buf); i++ {
		data := uint16(buf[i])
		data ^= _crc & 0xff
		data ^= (data & 0x0f) << 4
		_crc = ((data << 8) | (_crc >> 8)) ^ (data >> 4) ^ (data << 3)
	}
	return _crc
}
```

### The Initial Handshake / Heartbeat
To keep the connection alive, the client must periodically request information using an incrementing ID.
- **Request:** `{"id":2,"method":"get_info"}`
- Example Hex Generation: `ff aa 1c 00 7b 22 69 64 22 3a 32 2c 22 6d 65 74 68 6f 64 22 3a 22 67 65 74 5f 69 6e 66 6f 22 7d 7e 22 fe`

## Current Status and Roadblocks
- We successfully intercepted the exact bytes the manufacturer binary `gklib` transmits when opening the port by using `strace -e trace=write,openat`.
- We developed `k3c/cmd/ace_client/main.go`, a standalone Go daemon that mimics Klipper's exact termios flags, dynamically generates JSON get_info queries, calculates the exact specific CRC algorithm, correctly wraps the packet in `FF AA ... FE` frames, and broadcasts it to the ACE module.
- We verified with `test_compare.go` that our standalone client's generated bytes are byte-for-byte identical to the manufacturer's packets.
- **Current Blockers:** Even when transmitting the syntactically perfect initial sequence, our standalone `ace_client` sometimes still triggers the ACE device's watchdog disconnect (resulting in `input/output error` loop).

## Next Steps
1. **Timing Analysis:** Review if there's a specific wait time needed between opening the FD and transmitting the first byte, or if the transmission timing of subsequent heartbeats is too slow/fast.
2. **Missing Initialization Command:** Identify if `gklib` uses IOCTLs to flush the buffer `TCFLSH` BEFORE sending data, which we might be missing.
3. **Response Verification:** Check if our read loop is draining the responses fast enough, or if the ACE drops connection when its TX buffer gets full.
4. Continue extracting trace logs from an authentic `gklib` startup to ensure no earlier packets, different Baud resets, or different `/dev/` targets (e.g. `ttyS3/ttyS5`) are being used to prime the hardware before it pivots to `/dev/ttyACM0`.

## Critical Connection Stabilization (Fix for Hardware Watchdog / TTY Drops)
Through cross-referencing with the [Antitriad Kobra-S1/ACEPRO](https://github.com/Antitriad/Kobra-S1/tree/main/ACEPRO) repository and subsequent native Go testing, the exact requirements to stop the ACE hardware from aggressively dropping the USB (`/dev/ttyACM0`) connection were isolated:

1. **Immediate I/O Flush on Open**:
   When the ACM port is opened, stale bytes might exist on the bus. This confuses the ACE microprocessor, triggering a failure before the protocol can handshake. 
   **Fix**: Immediately after the `unix.TCSETS` termios configuration, flush the buffers:
   ```go
   unix.IoctlSetInt(fd, unix.TCFLSH, unix.TCIOFLUSH)
   ```

2. **The Minimum Heartbeat (3-Second Rule)**:
   The hardware disconnects unless valid framed protocol commands hit its buffer every few seconds. Valid keepalives consist of:
   - Initial Handshake: `{"id":1, "method":"get_info"}`
   - Polling Loop: `{"id":2, "method":"get_status"}` repeated continuously (e.g., every 1-2 seconds).

These two components—**clearing stale state on open (`TCIOFLUSH`) + maintaining continuous valid framing loop**—completely eliminate the previous mystery `USB disconnect` issues and provide a 100% stable Anycubic ACE Pro connection strictly in Go.
